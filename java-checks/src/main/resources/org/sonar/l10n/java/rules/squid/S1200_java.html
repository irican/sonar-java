根据Robert C. Martin在他的《面向对象设计原理》一书中介绍的单一职责原则，一个类应该只有一个职责：

如果一个类有多个职责，那么这些职责功能就会耦合起来。

一项功能的变更可能会损害或抑制类与其他类的接触（meet）能力。

这种耦合导致脆弱的设计，在更改时以意想不到的方式被破坏。

依赖于许多其他类的类往往聚合了太多的功能，应该划分为几个较小的类。

嵌套类依赖项不计为外部类的依赖项。

不合规的代码示例

阈值为5：

class Foo {                        // 不合规 - Foo 取决于太多的类： T1， T2， T3， T4， T5， T6 和 T7
  T1 a1;                           // Foo 与 T1 耦合
  T2 a2;                           // Foo 与 T2 耦合
  T3 a3;                           // Foo 与 T3 耦合

  public T4 compute(T5 a, T6 b) {  // Foo 与 T4，T5 和 T6 耦合
    T7 result = a.getResult(b);    // Foo 与 T7 耦合
    return result;
  }

  public static class Bar {        // 合规 - Bar 依赖于两个类：T8 和 T9
    T8 a8;
    T9 a9;
  }
}
