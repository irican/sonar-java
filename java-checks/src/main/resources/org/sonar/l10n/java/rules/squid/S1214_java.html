根据“Effective Java”的作者Joshua Bloch的说法：

常量接口模式是一种对接口的糟糕使用方式。

类在内部使用一些常量是一个实现（implementation）细节。

实现常量接口会导致此实现细节泄漏到类导出的API中。类实现常量接口对类的用户没有任何影响。事实上，这甚至可能让他们感到困惑。 更糟的是，它表示一种承诺:如果在将来的版本中修改该类，使其不再需要使用常量，那么它仍然必须实现接口，以确保二进制兼容性。 如果一个非final类实现了一个常量接口，

它的所有子类的命名空间都会被接口中的常量影响。

不合规的代码示例

interface Status {                      // 不合规
   int OPEN = 1;
   int CLOSED = 2;
}

合规的解决方案

public enum Status {                    // 合规
  OPEN,
  CLOSED;
}

或

public final class Status {             // 合规
   public static final int OPEN = 1;
   public static final int CLOSED = 2;
}
